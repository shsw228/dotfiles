# Global Agent Instructions

このファイルはすべてのプロジェクトで読み込まれる基本ガイドです。
汎用的なルールを記載し、案件固有の要件は各リポジトリの指示ファイルに記載してください。

## 目的
- 安全で再現可能な作業を優先する
- 変更理由と影響範囲を明確にする
- ユーザーの意図確認を短い往復で済ませる

## コミュニケーション
- 不明点は最小限の質問で確認する
- 進め方の選択肢がある場合は短く提示する
- できない／不確かなことは明確に伝える
- 内部の思考は英語で行い、ユーザーへの応答は日本語で行う

## 作業方針
- まず現状確認（ファイル/構成/依存関係）を行う
- 既存の規約やスタイルを尊重する
- 影響が大きい変更は段階的に提案する
- Python を実行する際はグローバル環境を直接汚さず、原則として `uv run` など `uv` 経由を用いる

## 変更の扱い
- 破壊的操作は事前確認する
- 既存の未コミット変更は勝手に消さない
- 変更箇所は簡潔に説明する

## コミット・PR規則
- コミットは依頼がある場合のみ行う
- 既存のコミット規約があればそれに従う
- コミットメッセージとGitHubのコメントは日本語で記載する
- コミットメッセージ形式は `[種類] 変更内容の説明` を用いる
- 種類は次から選ぶ: `feat`（新機能）, `fix`（バグ修正）, `docs`（ドキュメント）, `style`（見た目/整形）, `refactor`（振る舞い変更なしの改善）, `test`（テスト）, `chore`（雑務）, `perf`（性能改善）, `build`（ビルド/依存）, `ci`（CI設定）
- PRタイトルも日本語で記載し、変更内容を簡潔に説明する

## テスト/検証
- 可能なら推奨テストを案内する
- 実行できない場合は理由を伝える

## 作業履歴
- 手動の作業ログファイルは残さず、変更履歴は `git` のコミット履歴のみで管理する

## 指示ファイルの使い分け
- リポジトリ内の指示ファイルには、そのリポジトリに限って必要な情報、制約、コマンド、規約だけを書く
- 複数リポジトリに共通する作業方針やユーザーの一般的な好みは、グローバル指示に寄せる
- リポジトリ内の指示ファイルを編集する前に、その指示が本当にリポジトリ固有かを確認する

## PR作成・更新
- PR作成または更新の前に、必ず `git diff --name-status origin/main...HEAD` でベースブランチとの差分を確認する
- PRタイトルと本文は実際の差分に含まれる変更だけを元に書き、以前のPR説明を安易に流用しない
- 変更がPRに載っていると伝える前に、`gh pr view` または `gh pr list` でPRがopenで現在ブランチを指していることを確認する

## 出力形式
- 冗長になりすぎない
- コードやコマンドは必要最小限に
- 参照箇所はパスで示す

## 例外/補足
- プロジェクト固有のルールは都度追記する
- このガイドと矛盾する指示があれば、そちらを優先する
